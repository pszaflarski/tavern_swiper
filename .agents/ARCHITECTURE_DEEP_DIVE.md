# 🏰 Architectural Deep Dive: Multi-User Matches

This document provides a technical walkthrough of how "Matches" are formed and verified in the Tavern Swiper ecosystem. Use this as a reference when modifying the Discovery or Swipes services.

## 🚀 The Match Lifecycle

Matches are a distributed event involving three primary microservices. No single service owns the entire state; they interact via REST APIs as follows:

1.  **Identity Creation (`Profiles` Service)**:
    *   A user must forge an identity before entering the Tavern.
    *   **Logic**: `POST /profiles/` saves the profile and attributes to the `profiles` Firestore database.
    *   **Agent Tip**: The frontend must set `activeProfileId` immediately after creation to ensure the Discovery feed loads the correct context.

2.  **Discovery Feed (`Discovery` Service)**:
    *   The Tavern feed is generated by filtering for all profiles *except* those belonging to the current `user_id` and those already swiped by the `activeProfileId`.
    *   **Logic**: `GET /discovery/feed/{profile_id}`.
    *   **Dependency**: Discovery queries the `Profiles` service to get candidate details and the `Swipes` service to check for existing swipes.

3.  **The Swipe (`Swipes` Service)**:
    *   When a user likes a hero, a swipe record is created.
    *   **Logic**: `POST /swipes/` with `swiper_profile_id`, `swiped_profile_id`, and `direction='right'`.
    *   **Match Check**: The `Swipes` service automatically checks if a reciprocal RIGHT swipe exists from the target profile to the swiper. If it does, a **Match** record is created in the `swipes` Firestore database.

4.  **Match Retrieval**:
    *   **Logic**: `GET /swipes/matches/{profile_id}` returns all mutual matches for that specific identity.

## 🧪 Testing and Verification

Because the system is "Zero-Trust," we never assume the UI "knows" a match happened. We always verify the state directly via the backend.

### **E2E Strategy**
*   **Multi-Context Orchestration**: We use Playwright to simulate two independent users (A and B) simultaneously.
*   **UI Anchors**: The `index.tsx` contains `swipe-left-button` and `swipe-right-button` `testIDs` for reliable agent interaction.
*   **REST Validation**: After User A and B both swipe RIGHT, the test suite calls the `Swipes` REST API to confirm the match record exists. This ensures the backend trigger logic is firing correctly.

### **Common Gotchas**
*   **Timing**: Screen transitions in React Native Web can be slower than Playwright's default execution. Always use `.waitFor({ state: 'visible' })` on profile names before asserting.
*   **Authorization**: Every API call requires a Bearer Token. In E2E tests, we capture these tokens from the outgoing network requests during the login/signup flow.
